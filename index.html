Discovery & Strategy

Conducted competitive analysis of Chrome, Safari, Edge, and Firefox tab grouping implementations. Identified key patterns: visual grouping, drag-and-drop organization, and persistent group state.

Gathered user feedback from community forums and support channels. Power users with 20+ tabs consistently reported frustration with finding specific tabs. Usage data showed that these power users had 3x higher churn than average users.

Built business case demonstrating that tab grouping was becoming table stakes in the browser category. Presented competitive gap analysis to leadership (Design Lead, Product Lead, CEO, and engineering leads).

Secured buy-in by proposing phased rollout: ship iOS first to validate with real metrics, then decide on Android investment. This de-risked the initiative and allowed us to learn fast. Research directly informed the decision to focus on flexibilityâ€”supporting both bulk organization and spontaneous grouping.

Design Decisions & Rationale

Designed two interaction paths after observing that users organize tabs in two distinct ways: bulk organization sessions (reorganizing many tabs at once) and opportunistic grouping (moving one tab while browsing).

Batch selection serves power users who need to organize 20+ tabs quickly. Select multiple tabs with checkboxes, then group with one action. Drag-and-drop serves casual users making quick, spontaneous moves between existing groups. This flexibility accommodates different user types and contexts.

Most browsers make users exit the tab manager to switch between normal and private modes. This creates unnecessary friction. By embedding the mode switcher directly in the tab manager, we reduced this multi-step process to a single tap, keeping users in their flow.

Gave Sync mode (cross-device tab syncing, unique to Aloha) equal prominence with Normal and Private. This positioning elevated sync from a hidden feature to a first-class capability, supporting our power user value proposition.

Created distinct visual treatment for private mode using a darker color palette and unique iconography. This clear differentiation helps users feel confident they're in a secure browsing contextâ€”critical for a privacy-focused browser. By giving private mode equal prominence and full grouping support, we reinforced Aloha's core privacy-first brand promise.

Made groups collapsible with clear visual hierarchy. Group headers show tab count and support quick actions: rename, color picker, and delete. Color coding helps users create mental models ("work" is blue, "shopping" is green, etc.).

Implementation Approach

Shipped iOS first to validate the grouping concept with real user metrics before investing in full Android implementation. This strategy de-risked the featureâ€”if grouping flopped on iOS, we'd save months of Android development.

iOS users adopted grouping quickly: 43% created at least one group within the first week, and Day 1 retention improved by 1.2%. This validated the concept and greenlit Android development.

Collaborated closely with iOS and Android engineers throughout implementation. Ensured gestures felt platform-native: iOS uses swipe actions for quick group actions, Android uses long-press. Same functionality, platform-appropriate execution.

Iterated post-launch when initial tab card shapes caused visual issues across landscape and portrait orientations. Final design simplified the visual system while improving performance.

For Android, partnered with the engineering lead to design UI foundations matching Material Design patterns. Android engineering flagged that rich tab previews caused performance issues on devices with limited RAM. Older devices would lag or crash when users had 30+ tabs open.

Rather than compromise the experience for all users or abandon visual previews entirely, I designed a user-controlled toggle between preview mode and list mode. List mode uses text-only representation, reducing memory overhead by 60% while maintaining full grouping functionality.

Positioned the toggle as a power user feature rather than a technical fallback. The framing was "choose your view" rather than "fix performance problems." This preserved user agency and avoided making legacy device users feel like second-class citizens.

Unexpectedly, users with newer devices also switched to list mode when managing 50+ tabsâ€”they discovered it was faster for scanning large collections. The technical constraint became a legitimate feature serving different user needs and contexts.

Company pivoted to premium features before Android grouping shipped, but the UI improvements and architecture laid the foundation for future implementation.

Key Learnings

Phased approach (iOS-first) significantly reduced risk. Validating with real metrics before full cross-platform investment saved time and informed better decisions for Android.

Platform-specific design decisions (respecting iOS and Android conventions) led to better user adoption than forcing identical experiences would have.

Technical constraints can become features when framed correctly. The Android list view toggle became a power user tool, not a compromise.

The unconventional mode switcher approach paid off: power users who discovered it had 40% higher engagement with private browsing and 2x higher sync adoption. Sometimes breaking conventions delivers better outcomes.
